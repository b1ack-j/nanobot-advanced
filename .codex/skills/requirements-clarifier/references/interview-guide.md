# Interview Guide

Question bank organized by area. Use as inspiration -- adapt to context. Ask one question per turn.

## Goal and Value

- What concrete outcome do you want from this project?
- If this project succeeds perfectly, what changes for you?
- What is the single most important thing this must achieve?
- Why is this valuable now (vs 6 months ago or 6 months from now)?

## Background and Motivation

- What triggered this initiative?
- Have you tried anything similar before? What worked, what didn't?
- What must remain unchanged from current practice?
- Are there prior decisions or existing commitments that constrain this?

## Target Users / Stakeholders

- Who will use this directly?
- Who cares about the outcome but won't use it directly?
- Are there different user types with different needs?

## Scope

- What must be included in this phase?
- What is explicitly out of scope for now?
- What adjacent ideas should be deferred to later?
- If you had to cut one feature to ship faster, which would it be?

## Constraints

- What technical constraints are non-negotiable (language, platform, infra)?
- What budget or time constraints exist?
- Are there compliance or regulatory requirements?
- What risks are acceptable vs unacceptable?
- When constraints conflict, what's the priority order?

## Current State (Code Evidence)

- Which existing components already support this goal?
- What's clearly missing in the current codebase?
- Is there known technical debt that could block progress?
- What existing patterns should we follow or break?

## Desired End State and Success Criteria

- What does "done" look like for this phase?
- What behaviors must the solution consistently exhibit?
- What failure modes are most critical to avoid?
- What measurable criteria will prove success?
- How will you know this was worth building?

## Conflict Resolution Prompts

When contradictions arise:
- "Earlier you said X, but now Y -- which should we go with?"
- "These two requirements seem to conflict: A vs B. Which takes priority?"
- "Should we keep the original scope, or adjust based on what we've learned?"

## Best Practices

1. Start with Goal -- everything else follows from it
2. Ask the question you're most uncertain about first
3. If the user gives a vague answer, probe deeper on the same topic
4. Summarize understanding after every 2-3 questions
5. Make assumptions explicit -- "I'm assuming X, correct?"
6. Watch for scope creep -- gently redirect to current phase
